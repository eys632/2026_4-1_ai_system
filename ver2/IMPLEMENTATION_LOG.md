# 구현 및 실험 과정 기록

- 전주대 인공지능학과 웹(학과안내) + 복지서비스 PDF를 기반으로 RAG 구축
- **청킹 전략(기본 vs LLM 기반)** 비교 실험(LLM 청킹은 답변 LLM과 분리)
- 웹 UI에서 동일 질문에 대해 두 방식 답변을 동시에 출력
- 실행환경: `/home/a202192020/miniconda3/envs/ys_conda1_env` (가상환경 새로 만들지 않음)

## 1.1 구현 중 겪은 문제(시간순)

아래는 구현 과정에서 실제로 발생했던 문제를 “문제”로 명확히 적고, 발생/대응을 시간 흐름대로 정리한 것입니다.

1) 실행 환경 제약(새 가상환경 금지)
- 문제: 새 venv/conda를 만들지 말고 지정된 conda 환경(`/home/a202192020/miniconda3/envs/ys_conda1_env`)에서만 설치/실행해야 했음.
- 대응: 모든 설치/실행 절차를 해당 환경 기준으로 고정하고, README에 활성화 명령을 명시.

2) 웹 데이터 수집 중 일부 URL 404
- 문제: 수집 대상 중 `lab.do`, `admin.do`가 404로 실패.
- 영향: 수집 스크립트가 중단되거나(혹은 데이터 누락으로) 후속 단계가 불안정해질 수 있음.
- 대응: 수집 루프에서 예외 처리로 실패 URL은 스킵하고, 정상 페이지는 계속 수집되도록 수정.

3) PDF가 RAG에 포함되었는지 검증 필요
- 문제: PDF 로더를 붙였더라도 실제로 전처리 산출물과 인덱스에 반영됐는지 확인이 필요했음.
- 대응: `data/processed/documents.jsonl`에 PDF 문서가 포함됐는지 확인하고, `index/*/stats.json`의 문서/청크 수로 2차 확인.

4) 청킹 비교 실험을 위한 인덱스 분리 필요
- 문제: baseline 청킹과 LLM 청킹을 같은 인덱스에 섞으면 비교 실험이 무의미해짐.
- 대응: 인덱스를 `index/baseline/`와 `index/llm/`로 분리해 별도 구축/저장.

5) LLM 청킹 출력 포맷 불안정(JSON 파손 가능)
- 문제: LLM 청킹은 JSON 배열 형태의 청크 목록을 기대하지만, 모델이 형식을 깨거나 불필요 텍스트를 섞을 수 있음.
- 대응: JSON만 출력하도록 지시를 강화하고, 파싱 실패 시 baseline 청킹으로 폴백하여 전체 파이프라인이 중단되지 않게 함.

6) UI 요구사항(두 방식 답변 + 소요 시간 동시 표시)
- 문제: 같은 질문에 대해 두 답변을 나란히 출력하고, 각각 소요 시간을 표시해야 했음.
- 대응: Gradio에서 baseline/llm 각각 실행 시간을 측정해 결과 상단에 표시.

7) 특정 질문에서 출력이 이상해짐(역할 토큰 섞임/근거만 출력/중복)
- 문제(사례): “에너지 취약계층 지원” 질문에서
  - 답변에 `user`, `assistant` 같은 role marker가 섞이거나
  - 답변이 본문 없이 “근거:”만 나오거나
  - baseline 답변이 중복/반복되는 현상이 관찰됨.
- 대응:
  - 채팅 템플릿 적용을 보정하고, 출력 후처리로 role marker 제거
  - 생성 옵션을 안정화(예: `temperature=0.0`, 반복 억제 설정)
  - 프롬프트 규칙 강화(근거 밖 추측 금지, 중복 금지, (근거: …) 표기)
  - 에너지 관련 질문은 검색 후보를 확장(top-30)한 뒤 키워드 필터 + 카테고리 다양화(diversify)로 중복을 줄이는 post-retrieval 로직 적용

8) 검색 근거 디버깅(출력/확인이 불편)
- 문제: 검색 결과가 길거나 특수문자 이슈로 콘솔에서 확인이 불안정한 경우가 있었음.
- 대응: 근거를 확인 가능한 형태로 덤프/가공해 “실제로 어떤 청크가 선택됐는지”를 검증.

9) 서버 재시작/포트 점유 등 운영 이슈
- 문제: 변경사항 반영을 위해 앱을 재시작해야 하고, 간혹 포트(7860)가 점유될 수 있음.
- 대응: 종료(`Ctrl + C`) 및 재실행 절차를 README에 명시하고, 필요 시 포트 정리 방법도 안내.

10) GitHub 푸시 차단(GH007: private email)
- 문제: 레포를 교체하여 푸시하는 과정에서 커밋 author 이메일이 비공개 설정과 충돌해 GitHub가 푸시를 거절.
- 대응: author 이메일을 GitHub `noreply`로 변경 후 커밋을 amend하여 재푸시.

## 2. 데이터 수집/전처리
- 웹 크롤링: requests/bs4/lxml로 주요 텍스트 추출
    - 인사말/학과소개/교수진소개/강의실습실/행정안내 페이지 대상
    - `lab.do`, `admin.do` URL이 404 에러 발생 → try/except로 예외처리, 나머지 3개만 수집
- PDF: pypdf로 페이지별 텍스트 추출
- 산출물: `data/processed/documents.jsonl`, `summary.json`

## 3. 인덱스 구축
- baseline: 고정 크기+overlap 청킹
- llm: Qwen 0.5B로 의미 단위 청킹(JSON 배열), 실패 시 fallback
- 임베딩: E5(multilingual-e5-small), FAISS(IndexFlatIP)
- 인덱스 저장: `index/baseline/`, `index/llm/`

## 4. RAG/생성/프롬프트
- 답변 LLM: Qwen2.5-1.5B-Instruct
- 프롬프트 템플릿: 근거만 사용, 반복 금지, 항목별 정리, (근거: 1,2,...) 표기
- temperature=0.0, repetition_penalty=1.05 등 안정화 옵션 적용
- LLM 출력에서 role marker(user/assistant/system) 제거 후처리

## 5. UI/실험
- Gradio Blocks로 baseline/llm 답변 동시 출력, 소요 시간(ms) 표시
- PDF 포함 여부/문서/청크 통계로 검증
- 파이프라인 설명은 PIPELINE.md로 별도 작성

## 6. 품질 개선/디버깅
- "에너지 취약계층 지원" 질문에서 이상 출력(무관 항목/반복/user 토큰 등)
    - LLM 생성 템플릿/후처리/프롬프트/post-retrieval 필터/다양화 로직 순차적 개선
    - 에너지 질문은 top-30 후보 검색 후 키워드 필터+카테고리별 diversify
    - baseline 답변 반복은 일부 남아있으나, llm 쪽은 품질 개선 확인
- 서버 재시작 및 정상 동작 확인
